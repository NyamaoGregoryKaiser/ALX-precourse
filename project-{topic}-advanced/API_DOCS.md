# AppInsight API Documentation

The AppInsight backend exposes a comprehensive set of RESTful APIs for managing monitored applications, metrics, and ingesting/retrieving performance data. This documentation describes how to access and use the API.

## 1. Accessing the Interactive API Documentation (Swagger UI)

For an interactive and up-to-date view of all API endpoints, including request/response schemas, try-it-out functionality, and authentication workflows, please navigate to the **Swagger UI**:

**URL:** `http://localhost:8080/swagger-ui.html`

This interface is automatically generated by [SpringDoc OpenAPI](https://springdoc.org/) based on the Spring Boot application's code and annotations.

## 2. API Endpoints Overview

The API is structured around resource types: Authentication, Monitored Applications, Metrics, and Metric Data.

### Base URL

All API endpoints are prefixed with: `http://localhost:8080/api`

### Authentication and Authorization

*   **JWT-based Security**: All protected endpoints require a JSON Web Token (JWT) obtained from the `/api/auth/login` endpoint.
*   **Request Header**: Include the JWT in the `Authorization` header: `Authorization: Bearer <YOUR_JWT_TOKEN>`
*   **Roles**: Endpoints are secured with role-based access control (`ADMIN`, `USER`).

| Endpoint                                | Method | Description                                    | Authentication     | Roles               |
| :-------------------------------------- | :----- | :--------------------------------------------- | :----------------- | :------------------ |
| `/api/auth/register`                    | `POST` | Register a new user account.                   | None               |                     |
| `/api/auth/login`                       | `POST` | Authenticate user and get JWT token.           | None               |                     |

### Monitored Applications

Resources for managing the applications you want to monitor.

| Endpoint                                | Method | Description                                    | Authentication     | Roles               |
| :-------------------------------------- | :----- | :--------------------------------------------- | :----------------- | :------------------ |
| `/api/applications`                     | `GET`  | Retrieve a list of all monitored applications. | Required           | `ADMIN`, `USER`     |
| `/api/applications/{id}`                | `GET`  | Retrieve a single application by ID.           | Required           | `ADMIN`, `USER`     |
| `/api/applications`                     | `POST` | Create a new monitored application.            | Required           | `ADMIN`             |
| `/api/applications/{id}`                | `PUT`  | Update an existing application.                | Required           | `ADMIN`             |
| `/api/applications/{id}`                | `DELETE` | Delete an application.                         | Required           | `ADMIN`             |

### Metrics

Resources for defining performance metrics within a monitored application.

| Endpoint                                          | Method | Description                                    | Authentication     | Roles               |
| :------------------------------------------------ | :----- | :--------------------------------------------- | :----------------- | :------------------ |
| `/api/applications/{applicationId}/metrics`       | `GET`  | Get all metrics for a given application.       | Required           | `ADMIN`, `USER`     |
| `/api/applications/{applicationId}/metrics/{metricId}` | `GET`  | Get a single metric by its ID.                 | Required           | `ADMIN`, `USER`     |
| `/api/applications/{applicationId}/metrics`       | `POST` | Create a new metric for an application.        | Required           | `ADMIN`             |
| `/api/applications/{applicationId}/metrics/{metricId}` | `PUT`  | Update an existing metric.                     | Required           | `ADMIN`             |
| `/api/applications/{applicationId}/metrics/{metricId}` | `DELETE` | Delete a metric.                             | Required           | `ADMIN`             |

### Metric Data

Resources for ingesting and retrieving actual performance data points.

| Endpoint                                    | Method | Description                                    | Authentication     | Roles               |
| :------------------------------------------ | :----- | :--------------------------------------------- | :----------------- | :------------------ |
| `/api/metric-data/ingest`                   | `POST` | Ingest new metric data points.                 | `X-API-KEY` Header | `MONITORING_AGENT` (implicit) |
| `/api/metric-data/{metricId}`               | `GET`  | Retrieve metric data for a metric by time range.| Required           | `ADMIN`, `USER`     |
| `/api/metric-data/{metricId}/paginated`     | `GET`  | Retrieve paginated metric data for a metric.   | Required           | `ADMIN`, `USER`     |

## 3. Data Transfer Objects (DTOs)

See the `src/main/java/com/appinsight/appinsight/dto` package for detailed request and response payload structures. Key DTOs include:

*   `AuthRequest`, `AuthResponse`, `RegisterRequest`
*   `MonitoredApplicationDTO`
*   `MetricDTO`
*   `MetricDataRequest`

## 4. Error Handling

The API follows a consistent error response format. When an error occurs (e.g., `400 Bad Request`, `401 Unauthorized`, `403 Forbidden`, `404 Not Found`, `500 Internal Server Error`), the response body will typically contain:

```json
{
  "timestamp": "2023-10-27T14:30:00.123456",
  "message": "A human-readable error message.",
  "details": "More specific details about the error or the request context."
}
```

For validation errors (`400 Bad Request` due to `@Valid` annotations), the `message` might be "Validation Failed" and `details` will contain a string representation of specific field errors.

## 5. Rate Limiting

Certain endpoints, particularly `/api/metric-data/ingest`, are configured with rate limiting to protect against abuse and ensure stability. If a client exceeds the allowed request rate, the API will respond with a `429 Too Many Requests` HTTP status. The `Retry-After` header might also be present, indicating when the client can retry.

## 6. Caching

The API leverages in-memory caching (Caffeine) for frequently accessed `MonitoredApplication` and `Metric` data (`GET` by ID). This improves response times and reduces database load. Updates and deletions automatically evict relevant cache entries.
