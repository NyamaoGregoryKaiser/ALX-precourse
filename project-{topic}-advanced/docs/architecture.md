# Mobile Task Manager Backend Architecture Documentation

This document describes the architectural design and principles of the Mobile Task Manager Backend.

---

## 1. Overview

The backend is designed as a **monolithic service** built with Node.js and TypeScript, following a **layered architecture** pattern. It provides RESTful APIs for a mobile task management application, handling user authentication, task management, and category organization. The system prioritizes scalability, security, and maintainability.

## 3. Key Components and Layers

### a. Controller Layer (`src/modules/*/controller.ts`)

*   **Responsibility:** Handles incoming HTTP requests, extracts request parameters (body, query, params), delegates business logic to the Service Layer, and sends back HTTP responses.
*   **Characteristics:** Thin, focuses on request/response handling, minimal logic.
*   **Technologies:** Express.js `Request`, `Response`, `NextFunction`.

### b. Service Layer (`src/modules/*/service.ts`)

*   **Responsibility:** Contains the core business logic. It orchestrates operations, applies validation (beyond schema validation), interacts with the Repository Layer for data access, and implements domain-specific rules.
*   **Characteristics:** Business-focused, transactional logic, independent of HTTP specifics.
*   **Technologies:** Pure TypeScript functions, interacts with `repository.ts` and `utils/`.
*   **ALX Principles:** This layer directly embodies problem-solving and algorithm design. For example, ensuring unique category names per user involves specific checks, and task retrieval includes logic for filtering, sorting, and pagination.

### c. Repository Layer (`src/modules/*/repository.ts`)

*   **Responsibility:** Abstracts data access operations. It provides an interface for the Service Layer to interact with the database without knowing the underlying ORM or database technology.
*   **Characteristics:** Focused on CRUD operations, maps domain objects to database entities.
*   **Technologies:** Prisma Client (`src/database/prisma-client.ts`).

### d. Database Layer (`src/database/prisma-client.ts`, `prisma/schema.prisma`, `prisma/migrations/`)

*   **Responsibility:** Manages data persistence.
*   **Technologies:**
    *   **PostgreSQL:** Relational database for structured data storage.
    *   **Prisma ORM:** Provides a type-safe API for database queries, schema definition (`schema.prisma`), and database migrations (`prisma/migrations`). Prisma Client (`src/database/prisma-client.ts`) ensures a singleton instance and logging for development.

### e. Middleware Layer (`src/middleware/`)

*   **Responsibility:** Intercepts and processes requests before they reach the controllers or after controllers but before sending a response.
*   **Key Middlewares:**
    *   **Authentication (`auth.middleware.ts`):** Verifies JWTs, attaches user info to `req.user`, and handles authorization (role-based).
    *   **Validation (`validation.middleware.ts`):** Uses Zod to validate request bodies, query parameters, and path parameters against defined schemas.
    *   **Error Handling (`error-handler.middleware.ts`):** Catches all errors (both operational `AppError` and unexpected errors), logs them, and sends a consistent, user-friendly response.
    *   **Logging (`logging.middleware.ts`, Morgan):** Records details about incoming requests and responses.
    *   **Rate Limiting (`rate-limit.middleware.ts`):** Protects against abuse by limiting requests per IP address.
    *   **Security (`helmet`):** Sets various HTTP headers for security.
    *   **CORS (`cors`):** Handles Cross-Origin Resource Sharing.

### f. Utility Layer (`src/utils/`)

*   **Responsibility:** Contains reusable helper functions and classes that are not directly tied to business logic or infrastructure.
*   **Key Utilities:**
    *   **`AppError` (`app-error.ts`):** Custom error class for handling operational errors gracefully.
    *   **`jwt` (`jwt.ts`):** Functions for signing and verifying JWTs.
    *   **`password-hasher` (`password-hasher.ts`):** Bcrypt-based password hashing and comparison.
    *   **`redis-client` (`redis-client.ts`):** Redis client instance and helper functions for caching.
    *   **`logger` (`logger.ts`):** Winston-based structured logging utility.

### g. Configuration Layer (`src/config/`)

*   **Responsibility:** Manages environment variables and application constants.
*   **Technologies:** `dotenv` (`env.ts`), TypeScript interfaces for strong typing.

## 4. Design Principles

*   **Separation of Concerns:** Each layer and module has a distinct responsibility, improving modularity and maintainability.
*   **Loose Coupling:** Components are designed to be as independent as possible, reducing interdependencies. The Service layer doesn't know *how* data is stored, only *that* it can be accessed via the Repository.
*   **High Cohesion:** Elements within a module or layer are strongly related and work together to achieve a single purpose.
*   **DRY (Don't Repeat Yourself):** Reusable utilities (e.g., password hashing, JWT handling, error formatting) are centralized.
*   **Robust Error Handling:** Distinguishes between operational errors (user input, external service issues) and programming errors (bugs) to provide appropriate responses and prevent crashes.
*   **Security by Design:** Includes authentication, authorization, input validation, rate limiting, and security headers from the start.
*   **Scalability:** Stateless API (JWTs), use of PostgreSQL and Redis, and containerization with Docker supports horizontal scaling.
*   **Testability:** The layered architecture makes it easier to write unit, integration, and E2E tests by allowing mocking of dependencies between layers.
*   **Type Safety:** Extensive use of TypeScript ensures type consistency and catches errors at compile time, reducing runtime bugs.

## 5. Data Flow (Request Lifecycle)

1.  **Client Request:** A mobile client sends an HTTP request to the API Gateway/Load Balancer (if applicable), which forwards it to the Express application.
2.  **Express App Entry (`server.ts` -> `app.ts`):** The request first hits the main `app.ts` file.
3.  **Global Middlewares:**
    *   **Security:** `helmet` applies security headers.
    *   **CORS:** `cors` handles cross-origin requests.
    *   **Body Parsing:** `express.json()` and `express.urlencoded()` parse the request body.
    *   **Logging:** `morgan` and `requestLogger` log request details.
    *   **Rate Limiting:** `apiRateLimiter` checks request limits.
4.  **Authentication Middleware (`authenticateToken`):** If the route is protected, this middleware validates the JWT in the `Authorization` header. If valid, it decodes the user information and attaches it to `req.user`. If invalid or missing, it short-circuits with a 401 Unauthorized error.
5.  **Authorization Middleware (`authorizeRoles`):** (If role-based access is required for a route) Checks if `req.user` has the necessary role. If not, it short-circuits with a 403 Forbidden error.
6.  **Validation Middleware (`validate`):** Zod schemas validate the request parameters (body, query, params). If validation fails, a 400 Bad Request error is thrown.
7.  **Controller:** The request reaches the appropriate controller method (e.g., `taskController.createTask`). The controller extracts request data.
8.  **Service:** The controller calls a method in the Service Layer (e.g., `taskService.createTask`). The service contains the core business logic, performs additional validations, and makes decisions.
9.  **Caching (Redis):** The Service Layer might first check Redis for cached data (e.g., `cacheGet`). If a cache hit, it returns data directly. If not, it proceeds to the database.
10. **Repository:** The Service Layer calls a method in the Repository Layer (e.g., `taskRepository.createTask`) to interact with the database.
11. **Database (PostgreSQL via Prisma):** The Repository uses Prisma Client to execute database queries.
12. **Response back up:** Data flows back from the database to the Repository, then to the Service, then to the Controller.
13. **Caching (Redis):** If data was fetched from the database, the Service Layer might cache it in Redis (`cacheSet`) before returning.
14. **Controller Response:** The Controller formats the data and sends an HTTP response to the client.
15. **Error Handling:** If any error occurs at any stage, it is caught by `globalErrorHandler` middleware, which logs the error and sends a standardized error response to the client.

This layered architecture provides a clear separation of concerns, making the codebase easier to understand, maintain, and extend for future features.
