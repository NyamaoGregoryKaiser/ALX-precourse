# Mobile Task Manager Backend - API Documentation

This document serves as a guide to accessing and understanding the API documentation generated by FastAPI.

FastAPI automatically generates interactive API documentation based on the OpenAPI specification (formerly Swagger). This allows you to explore endpoints, understand request/response schemas, try out API calls directly from your browser, and see example requests/responses.

## Accessing the Documentation

Once the backend service is running (e.g., via `docker-compose up -d`), you can access the interactive documentation at the following URLs:

1.  **Swagger UI**:
    `http://localhost:8000/api/v1/docs`

2.  **ReDoc**:
    `http://localhost:8000/api/v1/redoc`

**Note**: Replace `localhost:8000` with the actual host and port where your API is deployed if it's different.

## Key Sections in the Documentation

The generated documentation is divided into sections, typically corresponding to API tags (e.g., "Auth", "Users", "Projects", "Tasks").

### 1. Authentication

*   **`POST /api/v1/auth/register`**: Register a new user.
    *   **Request Body**: `username`, `email`, `password`.
    *   **Response**: `User` object (excluding `hashed_password`).
*   **`POST /api/v1/auth/login`**: Obtain an access token.
    *   **Request Body**: `username` (can be email or username), `password` (form data).
    *   **Response**: `access_token`, `token_type` (e.g., `bearer`).
*   **`POST /api/v1/auth/test-token`**: Verify an access token.
    *   **Headers**: `Authorization: Bearer <your_access_token>`.
    *   **Response**: `User` object of the authenticated user.

### 2. Users (Requires Authentication)

*   **`GET /api/v1/users/me`**: Get details of the current authenticated user.
*   **`PUT /api/v1/users/me`**: Update details of the current authenticated user.
*   **`GET /api/v1/users/{user_id}`**: Get details of a specific user by ID. (Admin-only)
*   **`PUT /api/v1/users/{user_id}`**: Update a specific user by ID. (Admin-only)
*   **`DELETE /api/v1/users/{user_id}`**: Delete a specific user by ID. (Admin-only, cannot delete self)

### 3. Projects (Requires Authentication)

*   **`POST /api/v1/projects/`**: Create a new project.
    *   **Authorization**: Only the authenticated user can create a project, becoming its owner.
*   **`GET /api/v1/projects/`**: Retrieve a list of projects owned by the current user.
*   **`GET /api/v1/projects/{project_id}`**: Retrieve a specific project by ID.
    *   **Authorization**: Only the project owner or a superuser can view it.
*   **`PUT /api/v1/projects/{project_id}`**: Update a specific project by ID.
    *   **Authorization**: Only the project owner or a superuser can update it.
*   **`DELETE /api/v1/projects/{project_id}`**: Delete a specific project by ID.
    *   **Authorization**: Only the project owner or a superuser can delete it.

### 4. Tasks (Requires Authentication)

*   **`POST /api/v1/tasks/`**: Create a new task within a project.
    *   **Authorization**: Only the project owner or a superuser can create tasks in that project.
*   **`GET /api/v1/tasks/`**: Retrieve a list of tasks.
    *   **Query Parameters**: `project_id` (filter tasks by project) or `assignee_id` (filter tasks by assignee).
    *   **Authorization**:
        *   If `project_id` is specified, user must own the project or be a superuser.
        *   If `assignee_id` is specified, user must be the assignee or a superuser.
        *   If no filter, returns tasks assigned to the current user.
*   **`GET /api/v1/tasks/{task_id}`**: Retrieve a specific task by ID.
    *   **Authorization**: User must be the project owner, the task assignee, or a superuser.
*   **`PUT /api/v1/tasks/{task_id}`**: Update a specific task by ID.
    *   **Authorization**:
        *   Project owner or superuser: Full update capabilities.
        *   Task assignee: Can only update `status` and `description`.
*   **`DELETE /api/v1/tasks/{task_id}`**: Delete a specific task by ID.
    *   **Authorization**: Only the project owner or a superuser can delete it.

## How to use Swagger UI (Briefly)

1.  **Authorize**: Click the "Authorize" button (usually at the top right). Enter `Bearer <your_access_token>` in the value field and click "Authorize". This will apply your token to all subsequent requests until you close the browser or clear it.
2.  **Expand an Endpoint**: Click on an endpoint (e.g., `POST /api/v1/auth/login`) to expand its details.
3.  **Try it out**: Click the "Try it out" button.
4.  **Fill Parameters**: Enter required parameters in the request body or query fields.
5.  **Execute**: Click "Execute" to send the request.
6.  **View Response**: The response will be displayed below, including status code, response body, and headers.

This interactive documentation is an invaluable tool for understanding and interacting with the API during development and testing.
