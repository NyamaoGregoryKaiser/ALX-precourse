## 1. Deployment Strategy

VizFlow is designed for containerized deployment, making it highly portable and scalable. The recommended approach involves:

*   **Docker Images**: Packaging both the backend and frontend into Docker images.
*   **Container Orchestration**: Using Docker Compose for simple, single-host deployments or Kubernetes for scalable, multi-node deployments in cloud environments.
*   **CI/CD**: Automating the build, test, and deployment process using GitHub Actions or similar tools.

## 2. Prerequisites

*   **Git**: For cloning the repository.
*   **Docker Engine & Docker Compose**: Installed on your deployment server(s).
*   **Container Registry**: (e.g., Docker Hub, GitHub Container Registry, AWS ECR) for storing your Docker images.
*   **Cloud Provider Account**: (e.g., AWS, Azure, GCP) if deploying to a cloud-managed Kubernetes service or VMs.
*   **Secrets Management System**: (e.g., AWS Secrets Manager, Vault, Kubernetes Secrets) for managing sensitive environment variables in production.
*   **Domain Name and SSL Certificate**: For securing external access (HTTPS).

## 3. Configuration for Production

Before deploying, you **MUST** configure sensitive information and production-specific settings using environment variables.

### Backend Environment Variables

These variables are consumed by `vizflow-backend/src/main/resources/application.yml` and the `Dockerfile`.

*   `DB_HOST`: Hostname of the PostgreSQL database (e.g., `your-rds-endpoint.us-east-1.rds.amazonaws.com`).
*   `DB_PORT`: Port of the PostgreSQL database (default: `5432`).
*   `DB_NAME`: Database name (e.g., `vizflow_prod`).
*   `DB_USER`: Database username.
*   `DB_PASSWORD`: Database password.
*   **`JWT_SECRET`**: **CRITICAL**: A very long, complex, and truly random string (e.g., generated by `openssl rand -base64 64`). Never hardcode this.
*   `JWT_EXPIRATION_MS`: Token expiration in milliseconds (e.g., `3600000` for 1 hour, or `86400000` for 24 hours).
*   `RATE_LIMIT_ENABLED`: `true` or `false`.
*   `RATE_LIMIT_RPS`: Requests per second limit.
*   `SPRING_PROFILES_ACTIVE`: Set to `prod` for production-specific Spring configurations.

### Frontend Environment Variables

These variables are typically prefixed with `REACT_APP_` and are baked into the frontend build.

*   `REACT_APP_API_BASE_URL`: The public URL of your backend API (e.g., `https://api.yourdomain.com/api`).

## 4. Building Production Images

It's recommended to build and push your Docker images as part of your CI/CD pipeline. However, for manual builds:

1.  **Navigate to the backend directory:**
    ```bash
    cd vizflow/vizflow-backend
    ```
2.  **Build the backend image:**
    ```bash
    docker build -t your-registry/vizflow-backend:latest .
    ```
3.  **Navigate to the frontend directory:**
    ```bash
    cd ../vizflow-frontend
    ```
4.  **Build the frontend image:**
    *(Ensure you set `REACT_APP_API_BASE_URL` in an `.env` file or directly during the build step if not using `docker-compose build`)*
    ```bash
    docker build --build-arg REACT_APP_API_BASE_URL=https://api.yourdomain.com/api -t your-registry/vizflow-frontend:latest .
    ```
5.  **Push images to your container registry:**
    ```bash
    docker push your-registry/vizflow-backend:latest
    docker push your-registry/vizflow-frontend:latest
    ```
    (Replace `your-registry` with your actual registry path, e.g., `docker.io/yourusername`)

## 5. Container Orchestration

### Docker Compose Deployment (for Single-Host Production / Small Scale)

The `docker-compose.yml` file provided is suitable for a single-host deployment.

1.  **Prepare a `.env` file**: Place the production environment variables (from section 3) in a `.env` file in the root `vizflow/` directory.
2.  **Edit `docker-compose.yml`**:
    *   Change `image` names to reference your pushed images (e.g., `image: your-registry/vizflow-backend:latest`).
    *   Adjust `ports` if necessary (e.g., `80:80` for Nginx frontend).
    *   Ensure `backend`'s `DB_HOST` points to your *production* PostgreSQL instance (e.g., a managed cloud DB service endpoint, or `db` if PostgreSQL is also running in the same Docker Compose stack).
3.  **Deploy**:
    ```bash
    cd vizflow/
    docker compose pull # Pull the latest images
    docker compose up -d
    ```
4.  **Reverse Proxy / Load Balancer**: For production, it's highly recommended to place a reverse proxy (like Nginx, Caddy, or a cloud load balancer) in front of your frontend (and potentially backend) containers to handle:
    *   **SSL Termination**: Enforce HTTPS.
    *   **Domain Routing**: Map `yourdomain.com` to frontend, `api.yourdomain.com` to backend.
    *   **Load Balancing**: Distribute traffic if you run multiple instances.

### Kubernetes Deployment (Conceptual)

For highly available, scalable, and resilient deployments, Kubernetes is the industry standard.

1.  **Container Registry**: Ensure your Docker images are pushed to a registry accessible by your Kubernetes cluster.
2.  **Kubernetes Manifests**: Create Kubernetes YAML files for:
    *   **Deployments**: For backend (e.g., 3 replicas) and frontend (e.g., 3 replicas).
    *   **Services**: To expose deployments internally.
    *   **Ingress**: To expose services externally with domain routing and SSL termination.
    *   **Secrets**: To securely store `JWT_SECRET`, database credentials, etc.
    *   **Persistent Volume Claims (PVC)**: For PostgreSQL data persistence (if running DB in K8s).
    *   **ConfigMaps**: For non-sensitive configurations.
3.  **Managed Database**: For production, it's strongly recommended to use a managed PostgreSQL service (e.g., AWS RDS, Azure Database for PostgreSQL, GCP Cloud SQL) rather than deploying PostgreSQL directly in Kubernetes.
4.  **Apply Manifests**:
    ```bash
    kubectl apply -f k8s/backend-deployment.yaml
    kubectl apply -f k8s/frontend-deployment.yaml
    # ... and so on for services, ingress, secrets, etc.
    ```
5.  **Monitoring**: Integrate with Kubernetes monitoring tools (e.g., Prometheus, Grafana, built-in cloud monitoring).

## 6. CI/CD Pipeline Integration

The `.github/workflows/ci-cd.yml` file provides a basic CI/CD pipeline using GitHub Actions.

**Steps in the CI/CD Pipeline:**

1.  **`on: push`**: Triggered on pushes to `main` branch.
2.  **Build Backend**:
    *   Checks out code.
    *   Sets up JDK 17.
    *   Caches Maven dependencies.
    *   Builds the Spring Boot application using `mvn clean install -DskipTests`.
3.  **Run Backend Tests**:
    *   Starts a PostgreSQL Testcontainer.
    *   Runs `mvn test` to execute unit, integration, and API tests.
    *   Generates JaCoCo test coverage report.
4.  **Build & Push Docker Images (Conditional)**:
    *   This step would typically be enabled only after successful tests and a specific tag or merge to `main`.
    *   Logs in to Docker Hub (or other registry) using GitHub Secrets.
    *   Builds `vizflow-backend` and `vizflow-frontend` images.
    *   Tags and pushes images to the registry.
5.  **Deployment (Placeholder)**:
    *   This step depends heavily on your deployment environment (e.g., `kubectl apply` for Kubernetes, `ssh` and `docker compose pull && docker compose up -d` for Docker Compose).
    *   Requires secure credentials for your deployment target.

**GitHub Secrets for CI/CD:**

*   `DOCKER_USERNAME`: Your Docker Hub username.
*   `DOCKER_PASSWORD`: Your Docker Hub Access Token (not your actual password).
*   `KUBERNETES_KUBECONFIG` (if deploying to K8s)
*   `SSH_PRIVATE_KEY` (if deploying to a VM via SSH)

## 7. Monitoring and Logging

*   **Backend Logging**: Configured with Logback (`logback-spring.xml`) to output logs to console and a file (`logs/vizflow-backend.log`).
    *   In production, collect these logs using a centralized logging solution (e.g., ELK Stack - Elasticsearch, Logstash, Kibana; Datadog; Splunk).
*   **Application Metrics**: Spring Boot Actuator provides endpoints (`/actuator`) for monitoring application health, metrics (JVM, Tomcat, custom), and environment info.
    *   Expose `/actuator/prometheus` to scrape metrics with Prometheus.
    *   Integrate with Grafana for dashboarding.
*   **Infrastructure Monitoring**: Monitor CPU, memory, network I/O, disk usage of your Docker containers and underlying host machines.
*   **Database Monitoring**: Monitor PostgreSQL performance, query times, connections, and disk usage.

## 8. Security Best Practices

*   **Secrets Management**: Never hardcode secrets. Use environment variables, Docker secrets, Kubernetes secrets, or dedicated secrets management services.
*   **HTTPS Everywhere**: Enforce HTTPS for all external communication to prevent eavesdropping and data tampering. Use a valid SSL certificate.
*   **Least Privilege**:
    *   Database users should only have the minimum necessary permissions.
    *   Application containers should run with non-root users.
    *   IAM roles/policies for cloud resources should adhere to least privilege.
*   **Regular Updates**: Keep your dependencies (Spring Boot, Java, Node.js, libraries) and base Docker images updated to patch security vulnerabilities.
*   **Input Validation**: Rigorous server-side validation to prevent injection attacks (SQL, XSS, etc.).
*   **Security Scans**: Integrate SAST (Static Application Security Testing) and DAST (Dynamic Application Security Testing) tools into your CI/CD.
*   **Network Segmentation**: Isolate your database and backend from direct internet access, placing them behind firewalls and VPNs where possible.

## 9. Performance Testing

Before deploying to production, conduct performance tests to identify bottlenecks and ensure the system can handle expected load.

**Tools:**
*   **JMeter**: Open-source load testing tool.
*   **Gatling**: Scala-based load testing framework, good for complex scenarios.
*   **Locust**: Python-based, distributed user load testing tool.

**Example Scenario (JMeter):**

1.  **Login**: Simulate users logging in to obtain a JWT.
2.  **Browse Dashboards**: Fetch a list of dashboards.
3.  **View Dashboard**: Fetch details and data for a specific dashboard.
4.  **Interactive Filters**: (More advanced) Simulate applying filters to a visualization.

**JMeter Setup (Conceptual `login_and_fetch_data.jmx`):**

*   **Thread Group**: Define concurrent users, ramp-up time, and loop count.
*   **CSV Data Set Config**: For parameterized login credentials.
*   **HTTP Request Defaults**: Base URL, port.
*   **Login Request**: `POST /api/auth/login`. Extract JWT using a JSON Extractor.
*   **HTTP Header Manager**: Add `Authorization: Bearer ${JWT_TOKEN}`.
*   **Fetch Dashboard/Dataset Data**: `GET /api/datasets/{datasetId}/data`.
*   **Listeners**: Aggregate Report, Summary Report, View Results Tree (for debugging).

**Monitoring During Tests**:
*   Track CPU, memory, and network usage of backend containers/VMs.
*   Monitor database metrics (connections, query times, slow queries).
*   Observe application logs for errors or warnings.
